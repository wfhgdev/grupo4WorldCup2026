🌐 Internacionalización (i18n)

¿Qué es i18n en este proyecto?

i18n = internacionalización, pero aquí no es un selector de idioma del sitio. La app siempre muestra su interfaz en español. Lo que hace este sistema es traducir los datos en inglés que devuelve la API (football-data.org): nombres de equipos, estados de partido, nombres de fase, etc. Fuentes:

Orden de carga de los scripts

Los diccionarios se combinan antes de que i18n.js los necesite. El orden en el HTML de cada página importa:

<script src="../js/i18n/es/teams.js"></script>
<script src="../js/i18n/es/enums.js"></script>
<script src="../js/i18n/es/competition.js"></script>
<script src="../js/i18n/es/ui.js"></script>
<script src="../js/i18n/es/index.js"></script>  <!-- fusiona todo en window.__DICT -->
<script src="../js/i18n/i18n.js"></script>        <!-- lee window.__DICT y expone el objeto i18n -->

Cada archivo es/*.js define una variable global (window.__TEAMS_ES, window.__ENUMS_ES...). es/index.js los fusiona en window.__DICT y borra las variables temporales. Solo entonces i18n.js puede leer window.__DICT y ofrecer sus funciones.

⚠️ Error común: cargar i18n.js antes que los diccionarios — window.__DICT estaría vacío y todas las traducciones devolverían la clave sin traducir.

📖 Para principiantes: conceptos básicos

  1. i18n — abreviatura de "internacionalización" (18 letras entre i y n). Sistema de traducción sin duplicar código.
  2. Diccionario (dictionary) — objeto donde clave = código, valor = traducción.
    { "BRA": "Brasil", "ARG": "Argentina" }
  3. NAME_TO_TLA — diccionario inverso: nombre completo → código FIFA.
    { "Brazil": "BRA", "Argentina": "ARG" }
  4. Función de traducción t(key) — busca una clave en el diccionario. Si no la encuentra, devuelve la misma clave.
  5. 4 archivos de diccionario: teams.js (equipos), enums.js (estados), competition.js (nombres de torneos), ui.js (textos de interfaz).
💡 Consejo: empieza con teams.js y enums.js — cubren el 80% de las traducciones visibles.

Cómo funciona

Un diccionario asocia claves a traducciones:

window.__DICT = {
  BRA: 'Brasil',
  ARG: 'Argentina',
  ESP: 'España'
};

function t(key) {
  return dict[key] || key;
}

t('BRA'); // "Brasil"

Funciones de formato y traducción (i18n.js)

Además de t() y translateTeam(), i18n.js expone funciones que usan todas las páginas de datos:

  • formatDate(dateStr) — convierte una fecha ISO de la API a formato largo en español (toLocaleDateString('es-ES', ...)).
  • formatTime(dateStr) — igual, pero solo hora y minutos.
  • getRelativeDay(dateStr) — devuelve "Hoy", "Mañana" o "Ayer" comparando con la fecha actual; si no aplica, cae a fecha larga.
  • translateStatus(status) / translateStatusShort(status) — traducen SCHEDULED, IN_PLAY, FINISHED, etc. a texto en español (largo o corto para badges).
  • translateStage(stage) — traduce el nombre de fase (GROUP_STAGE, LAST_16, QUARTER_FINALS...) buscando claves STAGE_* en el diccionario.
i18n.formatDate('2026-06-15T18:00:00Z');   // "lunes, 15 de junio de 2026"
i18n.translateStatus('IN_PLAY');            // "En Juego"
i18n.translateStage('QUARTER_FINALS');      // "Cuartos de Final"

🌐 Internationalization (i18n)

📖 For beginners: basic concepts

  1. i18n — abbreviation for "internationalization" (18 letters between i and n). A translation system without duplicating code.
  2. Dictionary — an object where key = code, value = translation.
    { "BRA": "Brazil", "ARG": "Argentina" }
  3. NAME_TO_TLA — reverse dictionary: full name → FIFA code.
    { "Brazil": "BRA", "Argentina": "ARG" }
  4. Translation function t(key) — looks up a key in the dictionary. If not found, returns the key itself.
  5. 4 dictionary files: teams.js (teams), enums.js (statuses), competition.js (tournament names), ui.js (interface texts).
💡 Tip: start with teams.js and enums.js — they cover 80% of visible translations.

What is i18n in this project?

i18n = internationalization, but here it is not a site language switcher. The app UI is always in Spanish. What this system does is translate the English data the API returns (football-data.org): team names, match statuses, stage names, etc. Sources:

Script load order

Dictionaries are merged before i18n.js needs them:

<script src="../js/i18n/es/teams.js"></script>
<script src="../js/i18n/es/enums.js"></script>
<script src="../js/i18n/es/competition.js"></script>
<script src="../js/i18n/es/ui.js"></script>
<script src="../js/i18n/es/index.js"></script>  <!-- merges into window.__DICT -->
<script src="../js/i18n/i18n.js"></script>        <!-- reads window.__DICT -->

Loading i18n.js before the dictionaries would leave window.__DICT empty, so every translation would fall back to the untranslated key.

How it works

A dictionary maps keys to translations:

window.__DICT = {
  BRA: 'Brazil',
  ARG: 'Argentina',
  ESP: 'Spain'
};

function t(key) {
  return dict[key] || key;
}

t('BRA'); // "Brazil"

Format and translation functions (i18n.js)

Besides t() and translateTeam(), i18n.js exposes helpers used across every data page:

  • formatDate(dateStr) — converts an ISO date from the API to a long Spanish date.
  • formatTime(dateStr) — same, but hour and minutes only.
  • getRelativeDay(dateStr) — returns "Today"/"Tomorrow"/"Yesterday" (in Spanish) relative to now, falling back to a long date.
  • translateStatus(status) / translateStatusShort(status) — translate SCHEDULED, IN_PLAY, FINISHED, etc.
  • translateStage(stage) — translates stage names (GROUP_STAGE, LAST_16, QUARTER_FINALS...) via STAGE_* dictionary keys.
